# PostgreSQLDatabase - Self-Service Database Provisioning

The `PostgreSQLDatabase` resource allows you to create and manage PostgreSQL databases in your namespace with a simple, declarative API.

## Quick Start

Create a PostgreSQL database with just a few lines:

```yaml
apiVersion: idp.rottler.io/v1alpha1
kind: PostgreSQLDatabase
metadata:
  name: my-app-db
  namespace: my-app
spec:
  compute: small
  storage: 2Gi
  version: "17"
  ha: false
```

Apply it:

```bash
kubectl apply -f my-database.yaml
```

Within a few minutes, you'll have a running PostgreSQL database with automatically generated credentials!

## What You Get

When you create a `PostgreSQLDatabase`, the platform automatically provisions:

✅ **PostgreSQL cluster** running in your namespace
✅ **Automatic credentials** with a secure random password
✅ **Connection secret** containing all connection details
✅ **Service endpoints** for read-write and read-only access
✅ **Monitoring** via Prometheus/Grafana (built-in)
✅ **High availability** (optional, with 3 instances and automatic failover)

## Specification

### Required Fields

| Field     | Type   | Values                                 | Description                        |
| --------- | ------ | -------------------------------------- | ---------------------------------- |
| `compute` | string | `small`, `medium`, `large`             | CPU and memory tier                |
| `storage` | string | e.g., `2Gi`, `10Gi`, `50Gi`            | Storage size with units (Mi or Gi) |
| `version` | string | `"13"`, `"14"`, `"15"`, `"16"`, `"17"` | PostgreSQL version                 |

### Optional Fields

| Field | Type    | Default | Description                                                  |
| ----- | ------- | ------- | ------------------------------------------------------------ |
| `ha`  | boolean | `false` | Enable high availability (creates 3 instances with failover) |

### Compute Tiers

Resources allocated per compute tier:

#### Homelab Environment

| Tier   | CPU   | Memory  | Use Case                  |
| ------ | ----- | ------- | ------------------------- |
| small  | 100m  | 512 MiB | Dev/test, low traffic     |
| medium | 500m  | 1 GiB   | Staging, moderate traffic |
| large  | 1000m | 2 GiB   | Production, high traffic  |

#### AWS Environment (EKS)

| Tier   | vCPU | Memory | Instance Type  | Use Case                |
| ------ | ---- | ------ | -------------- | ----------------------- |
| small  | 2    | 8 GiB  | db.m7g.large   | Dev/test                |
| medium | 4    | 16 GiB | db.m7g.xlarge  | Production workloads    |
| large  | 8    | 32 GiB | db.m7g.2xlarge | High-traffic production |

**Note**: On AWS, you can choose between:

- **CNPG** (self-managed): Cost-effective option, runs in your EKS cluster
- **RDS** (managed service): _Coming soon_ - fully managed with enhanced durability

## Connection Details

### Connection Secret

The platform automatically creates a secret named `{database-name}-superuser` in your namespace with the following keys:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-app-db-superuser
  namespace: my-app
type: kubernetes.io/basic-auth
data:
  username: cG9zdGdyZXM= # base64: postgres
  password: <auto-generated> # 32-character random password
  host: bXktYXBwLWRiLXJ3... # my-app-db-rw.my-app.svc.cluster.local
  port: NTQzMg== # base64: 5432
  dbname: cG9zdGdyZXM= # base64: postgres
  uri: cG9zdGdyZXNxbDovL... # postgresql://postgres:<pass>@<host>:5432/postgres
  jdbc-uri: amRiYzpwb3N0Z3Jlc3FsOi8v... # jdbc:postgresql://<host>:5432/postgres
```

### Accessing Credentials

**Option 1: Environment Variables in Deployment**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - name: app
          image: myapp:latest
          env:
            - name: DB_HOST
              valueFrom:
                secretKeyRef:
                  name: my-app-db-superuser
                  key: host
            - name: DB_PORT
              valueFrom:
                secretKeyRef:
                  name: my-app-db-superuser
                  key: port
            - name: DB_USER
              valueFrom:
                secretKeyRef:
                  name: my-app-db-superuser
                  key: username
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: my-app-db-superuser
                  key: password
            - name: DB_NAME
              valueFrom:
                secretKeyRef:
                  name: my-app-db-superuser
                  key: dbname
```

**Option 2: Connection URI**

Many frameworks (Node.js, Python, Ruby, etc.) accept a single `DATABASE_URL`:

```yaml
env:
  - name: DATABASE_URL
    valueFrom:
      secretKeyRef:
        name: my-app-db-superuser
        key: uri
```

**Option 3: JDBC (Java Applications)**

```yaml
env:
  - name: SPRING_DATASOURCE_URL
    valueFrom:
      secretKeyRef:
        name: my-app-db-superuser
        key: jdbc-uri
  - name: SPRING_DATASOURCE_USERNAME
    valueFrom:
      secretKeyRef:
        name: my-app-db-superuser
        key: username
  - name: SPRING_DATASOURCE_PASSWORD
    valueFrom:
      secretKeyRef:
        name: my-app-db-superuser
        key: password
```

### Service Endpoints

Your database is accessible via Kubernetes services:

- **Write endpoint**: `{database-name}-rw.{namespace}.svc.cluster.local`

  - Use for INSERT, UPDATE, DELETE operations
  - Always points to the current primary instance

- **Read endpoint** (HA only): `{database-name}-ro.{namespace}.svc.cluster.local`
  - Use for SELECT operations to distribute load
  - Load-balances across all replicas

Example:

```bash
# For database named "my-app-db" in namespace "my-app":
# Write: my-app-db-rw.my-app.svc.cluster.local:5432
# Read:  my-app-db-ro.my-app.svc.cluster.local:5432
```

## Status Information

Check your database status:

```bash
kubectl get postgresqldatabase my-app-db -n my-app
```

View detailed status:

```bash
kubectl get postgresqldatabase my-app-db -n my-app -o yaml
```

### Status Fields

| Field                        | Type    | Description                                      |
| ---------------------------- | ------- | ------------------------------------------------ |
| `phase`                      | string  | Current phase (e.g., "Cluster in healthy state") |
| `ready`                      | boolean | Whether the database is ready for connections    |
| `readyInstances`             | integer | Number of running instances (1 or 3)             |
| `connectionDetailsSecretRef` | string  | Name of the secret with connection details       |
| `endpoint`                   | string  | Primary (read-write) service name                |
| `readOnlyEndpoint`           | string  | Read-only service name (HA only)                 |
| `currentPrimary`             | string  | Name of the current primary instance             |

### Checking Status

```bash
# Is my database ready?
kubectl get postgresqldatabase my-app-db -n my-app -o jsonpath='{.status.ready}'

# How many instances are running?
kubectl get postgresqldatabase my-app-db -n my-app -o jsonpath='{.status.readyInstances}'

# What's the write endpoint?
kubectl get postgresqldatabase my-app-db -n my-app -o jsonpath='{.status.endpoint}'

# Which instance is the primary?
kubectl get postgresqldatabase my-app-db -n my-app -o jsonpath='{.status.currentPrimary}'
```

**Note**: During initialization (first 10-30 seconds), optional status fields like `endpoint`, `readOnlyEndpoint`, and `currentPrimary` may be empty while the CNPG cluster is starting up. Once the cluster is ready, these fields will be automatically populated.

## Examples

### Example 1: Small Development Database

Perfect for development and testing:

```yaml
apiVersion: idp.rottler.io/v1alpha1
kind: PostgreSQLDatabase
metadata:
  name: dev-db
  namespace: my-team-dev
spec:
  compute: small
  storage: 2Gi
  version: "17"
  ha: false
```

- Single instance (no HA)
- Minimal resources (100m CPU, 512 MiB RAM)
- Latest PostgreSQL version
- Provisions in ~2-3 minutes

### Example 2: Production Database with High Availability

For production workloads requiring uptime:

```yaml
apiVersion: idp.rottler.io/v1alpha1
kind: PostgreSQLDatabase
metadata:
  name: prod-db
  namespace: my-team-prod
spec:
  compute: medium
  storage: 20Gi
  version: "16"
  ha: true
```

- 3 instances with automatic failover
- 1 primary + 2 read replicas
- Synchronous replication
- Provisions in ~4-6 minutes

### Example 3: Large Database for High-Traffic Applications

Maximum resources for demanding workloads:

```yaml
apiVersion: idp.rottler.io/v1alpha1
kind: PostgreSQLDatabase
metadata:
  name: critical-db
  namespace: production
spec:
  compute: large
  storage: 100Gi
  version: "17"
  ha: true
```

- 3 instances (1 GiB CPU, 2 GiB RAM each)
- 100 GiB storage per instance
- Latest PostgreSQL version
- Suitable for high-traffic production applications

### Upgrading PostgreSQL Version

To upgrade to a new PostgreSQL version:

1. **IMPORTANT**: Backup your data first!
2. Update the `version` field in your PostgreSQLDatabase spec
3. Apply the change: `kubectl apply -f my-database.yaml`
4. The operator will perform a rolling upgrade

**Note**: Only upgrade one major version at a time (e.g., 15 → 16, not 15 → 17).

### Scaling Storage

To increase storage:

1. Update the `storage` field in your spec
2. Apply the change: `kubectl apply -f my-database.yaml`
3. The operator will expand the underlying PVCs

**Note**: Storage can only be increased, not decreased.

### Enabling High Availability

To convert a single-instance database to HA:

1. Update `ha: false` to `ha: true`
2. Apply the change
3. The operator will create 2 additional replicas

**Note**: This may cause a brief write interruption during the transition.

## Troubleshooting

### Database Not Ready

```bash
# Check the status
kubectl describe postgresqldatabase my-app-db -n my-app

# Check events
kubectl get events -n my-app --sort-by='.lastTimestamp'

# Check the underlying CNPG cluster
kubectl get cluster my-app-db -n my-app
kubectl describe cluster my-app-db -n my-app
```

Common issues:

- **Insufficient resources**: Check if your namespace has resource quotas
- **Storage issues**: Verify storage class is available and PVCs are bound
- **Image pull errors**: Check if container images are accessible

### Connection Failures

```bash
# Verify the secret exists
kubectl get secret my-app-db-superuser -n my-app

# Check the secret contents
kubectl get secret my-app-db-superuser -n my-app -o yaml

# Verify the service exists
kubectl get svc -n my-app -l cnpg.io/cluster=my-app-db

# Test connection from a pod
kubectl run -n my-app psql-test --rm -it --restart=Never \
  --image=postgres:17 \
  --env="PGHOST=my-app-db-rw" \
  --env="PGUSER=postgres" \
  --env="PGPASSWORD=$(kubectl get secret -n my-app my-app-db-superuser -o jsonpath='{.data.password}' | base64 -d)" \
  -- psql -c "SELECT version();"
```

### Checking Logs

```bash
# Check CNPG operator logs
kubectl logs -n cnpg-system -l app.kubernetes.io/name=cloudnative-pg --tail=100

# Check database instance logs
kubectl logs -n my-app my-app-db-1
```

### Slow Provisioning

Expected times:

- Single instance: 2-3 minutes
- HA (3 instances): 4-6 minutes

If taking longer:

- Check storage provisioning: `kubectl get pvc -n my-app`
- Check image pulls: `kubectl get pods -n my-app`
- Check node resources: `kubectl top nodes`

## Best Practices

### Security

1. **Use dedicated database users**: Don't use the superuser for applications

   - Create application-specific users with limited privileges
   - See [examples/app-using-database.yaml](examples/app-using-database.yaml) for an example

2. **Rotate credentials regularly**: The superuser password doesn't auto-rotate

   - Implement your own rotation strategy if needed

3. **Network policies**: Restrict database access to authorized pods only
   ```yaml
   apiVersion: networking.k8s.io/v1
   kind: NetworkPolicy
   metadata:
     name: allow-app-to-db
     namespace: my-app
   spec:
     podSelector:
       matchLabels:
         cnpg.io/cluster: my-app-db
     ingress:
       - from:
           - podSelector:
               matchLabels:
                 app: my-app
   ```

### Performance

1. **Use read replicas**: For HA databases, send read traffic to the read-only endpoint

   ```python
   # Python example with SQLAlchemy
   WRITE_DB = os.getenv("DB_WRITE_URL")  # Points to {name}-rw
   READ_DB = os.getenv("DB_READ_URL")    # Points to {name}-ro
   ```

2. **Size appropriately**: Start with `small`, scale up as needed

   - Monitor actual usage before choosing larger tiers
   - Storage can be expanded without downtime

3. **Use connection pooling**: For high-traffic applications
   - Implement pooling in your application (e.g., pgbouncer, connection pools in frameworks)

### Reliability

1. **Enable HA for production**: Set `ha: true` for production databases

   - Provides automatic failover in ~20-40 seconds
   - No data loss with synchronous replication

2. **Test failover**: Regularly test that failover works

   ```bash
   # Delete the primary pod to trigger failover
   kubectl delete pod my-app-db-1 -n my-app

   # Watch failover happen
   kubectl get cluster my-app-db -n my-app -w
   ```

3. **Monitor metrics**: Databases are automatically monitored
   - Check Grafana dashboards for database metrics
   - Set up alerts for disk usage, connection counts, etc.

### Cost Optimization

1. **Right-size your database**: Don't over-provision

   - Start small, scale up based on actual metrics
   - Use `small` for dev/test, `medium` for most production workloads

2. **Choose the right tier**:

   - **Homelab**: Only CNPG available
   - **AWS Dev**: Use CNPG (cost-effective)
   - **AWS Prod**: Use CNPG for most workloads, RDS for critical data (coming soon)

3. **Clean up unused databases**:

   ```bash
   # List all databases in your namespace
   kubectl get postgresqldatabase -n my-app

   # Delete unused databases
   kubectl delete postgresqldatabase old-db -n my-app
   ```

## Limitations

- **Backups**: Automated backups are not yet configured (coming soon)
- **Point-in-time recovery**: Not available in Phase 1 (planned)
- **Cross-region replication**: Not supported
- **Database size limits**: Limited by available storage in cluster
- **Extensions**: Only standard PostgreSQL extensions are available
- **Managed service option**: AWS RDS support coming in Phase 2
